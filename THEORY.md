# A Theory of obsidian-periodic-notes

## What this system is for

This plugin exists to solve one problem: making the file system behave like a calendar. Obsidian stores everything as Markdown files in folders. Users want to think in terms of dates — "today's note," "this week's note," "next month" — and the file system has no native concept of time. The plugin bridges this gap by establishing a convention: a file whose name matches a Moment.js date format string _is_ the note for that date, at a particular granularity. The system's job is to maintain this mapping, make it fast to query, and provide the affordances (commands, sidebar calendar, template rendering) that let users operate in date-space while the vault operates in path-space.

The core entities are few. A **granularity** is one of four time scales: day, week, month, year. A **note config** binds a granularity to its three parameters: a Moment.js format string (how dates become filenames), a folder (where those files live), and an optional template (what goes inside a new note). The **cache** is the live index that maps between the two worlds: given a file path, what date and granularity does it represent? Given a date and granularity, which file is it? Everything else — commands, the calendar UI, template rendering — is downstream of these two lookups.

## The organizing ideas

**The format string is the schema.** This is the most important design decision and the one that shapes everything else. Unlike a database where you'd store the date as metadata and the filename could be anything, this system derives the date _from_ the filename by parsing it against the configured format. The format string serves triple duty: it's the rule for generating new filenames (`date.format(format)`), the rule for recognizing existing files (`moment(basename, format, true)`), and implicitly the uniqueness constraint (two files whose basenames parse to the same `startOf(granularity)` date are in conflict). This is elegant but fragile — change the format and every existing file becomes invisible to the cache until renamed.

**Frontmatter is the escape hatch.** When the filename convention breaks down — perhaps because a user renamed a note — a frontmatter key matching the granularity name (e.g., `day: 2026-03-20`) can claim the file for that date. Frontmatter matches take strict priority: once a file is indexed via frontmatter, filename-based re-resolution skips it entirely (`resolve()` line 177: `if (existing && existing.match === "frontmatter") return`). This is an important asymmetry. Frontmatter is the override mechanism, not a parallel path.

**The cache is dual-indexed and self-healing.** `NoteCache` maintains two maps: `byPath` (file path → CacheEntry) and `byKey` (canonical key → CacheEntry), where a canonical key is `"${granularity}:${date.startOf(granularity).toISOString()}"`. The `startOf` normalization is load-bearing — it ensures that any moment within a week, month, or year collapses to the same key. The `set()` method enforces a one-to-one invariant: when a new entry arrives, it evicts any previous entry for the same canonical key (even if it's a different file) and any previous entry for the same file path (even if it was a different date). This means last-write-wins in conflicts, with no user-visible error. The `getPeriodicNote()` path is O(1); the tradeoff is that `findAdjacent()` — which navigates to the next or previous note of the same granularity — requires sorting all entries for that granularity on every call (issue #118).

**Settings are a plain object, not reactive state.** The v2 refactor made this explicit: `plugin.settings` is a `Settings` object, mutated directly and persisted via `saveData()`. There is no Svelte store wrapping it. When settings change, the plugin fires a custom workspace event (`periodic-notes:settings-updated`) on Obsidian's event bus, and interested parties — the cache, the calendar store — subscribe to that event. This is a deliberate choice to keep Svelte confined to the calendar sidebar and nowhere else.

**The calendar is a view into the cache, not a separate data structure.** The Svelte calendar components never query the vault directly. They go through `CalendarStore`, which wraps the plugin's cache lookups in a Svelte-compatible notification mechanism. `CalendarStore.store` is a `Writable<number>` — a counter whose value doesn't matter, only its changes. When a vault event might have changed the periodic note landscape, the counter bumps, and the `$effect` in `Calendar.svelte` re-computes a `FileMap`: a `Map<string, TFile | null>` that pre-resolves every day, week, month, and year cell that the current calendar view needs. This is the FileMap pattern — a single derivation replaces what would otherwise be 42+ individual store subscriptions (one per day cell, plus week/month/year). Child components like `Day.svelte` do a synchronous `$derived` lookup from this map using `fileMapKey()`.

**The reactivity bridge is the most subtle piece of the architecture.** Svelte 5 runes (`$derived.by()`) do not automatically track Svelte 4 store subscriptions. The `Calendar.svelte` component works around this with a specific pattern: an `$effect` that manually calls `.subscribe()` on the store and returns the unsubscribe function (lines 44–54). This is not a bug or a transitional hack — it's the documented interop pattern for using Svelte 4 stores inside Svelte 5 runes mode. Issue #102 tracks a future migration to pure runes, at which point the `CalendarStore.store` counter would become a `$state` signal and the bridge would disappear.

**Template rendering is layered by granularity.** `applyTemplate()` processes tokens in a fixed order: universal tokens (`{{date}}`, `{{title}}`, `{{time}}`), then granularity-specific tokens. Day notes get `{{yesterday}}`, `{{tomorrow}}`, and dynamic `{{date+1d:format}}`. Week notes get `{{monday:format}}` through `{{sunday:format}}`. Month and year notes get `{{month+1m:format}}` and `{{year-1y:format}}`. The key design choice: tokens for one granularity are left unreplaced in another granularity's template. `{{yesterday}}` in a month note template is preserved as literal text, not expanded. This is correct — it means templates are not accidentally granularity-polymorphic.

## The seams

**Between the plugin and Obsidian.** The plugin depends on several Obsidian APIs and conventions: `Plugin` lifecycle, `Component` for scoped event registration, `ItemView` for the sidebar, vault events for file changes, `metadataCache` for frontmatter, `normalizePath` for cross-platform paths, the global `window.moment` instance. It augments the `Workspace` type with two custom events (`periodic-notes:settings-updated` and `periodic-notes:resolve`) via `obsidian.d.ts`. The `periodic-notes:resolve` event is interesting — it fires when the cache successfully resolves a file, carrying the granularity and TFile. This is the plugin's public API surface for other plugins that want to react to periodic notes being identified.

**Between cache and template.** Template application is triggered _from inside the cache's resolve path_: when a newly created file has zero bytes (line 195 of cache.ts), the cache calls `applyTemplateToFile()`. This coupling is deliberate — the cache is the single place that knows "this file just became a periodic note," so it's the natural trigger point. But it means the cache has a dependency on the template module, which itself depends on format utilities. The dependency arrow is: `main → cache → template → format`. This is a DAG, not circular, but the cache is doing more than caching — it's also the creation-side orchestrator.

**Between CalendarView and Svelte.** `CalendarView` (an Obsidian `ItemView`) mounts the Svelte `Calendar` component and communicates with it through two exported functions: `tick()` (re-check whether today has changed) and `setActiveFilePath()` (highlight the active note). Svelte communicates back through callback props: `onHover`, `onClick`, `onContextMenu`. This is a clean boundary — the Obsidian layer handles file operations, navigation, and menus; Svelte handles rendering and user interaction. The CalendarView never touches the DOM that Svelte owns, and Svelte never calls Obsidian APIs directly (it goes through `CalendarStore` and callback props).

**Between format.ts and everything else.** `format.ts` is the only module with zero Obsidian imports. It was extracted specifically to be directly testable. This boundary is principled and load-bearing: any logic that can be expressed as a pure function of (settings, format strings, filenames) belongs here. The tests for `format.ts` import directly from it. The tests for `template.ts` and `cache.ts` cannot import their source modules (because those import `obsidian`), so they re-implement the logic locally — this is the drift risk tracked in issues #93 and #116.

**Between commands and the plugin.** Commands go through `plugin.openPeriodicNote()`, which is the single entry point for "navigate to a date." The distinction between `jumpToAdjacentNote` (go to the next/previous _existing_ note) and `openAdjacentNote` (go to the next/previous _date_, creating if needed) is important: the first uses `findAdjacent()` and can fail with a notice; the second uses date arithmetic and always succeeds by creating.

## What changes easily and what doesn't

**Easy changes:**

- Adding a new template token for an existing granularity — add a case in `applyTemplate()`.
- Adding a new settings field per granularity — add to `NoteConfig`, update `DEFAULT_CONFIG`, add a `Setting` control in `settings.ts`.
- Changing the calendar's visual layout — the Svelte components are self-contained and purely presentational.
- Adding new commands — `getCommands()` returns an array per granularity; the factory pattern accommodates this.

**Moderate changes:**

- Adding a fifth granularity (quarter was explicitly removed in v2). The `Granularity` union, `granularities` array, `DEFAULT_FORMAT`, `DEFAULT_CONFIG`, `DEFAULT_SETTINGS`, `granularityLabels`, `labels`, the settings UI loop, the command factory loop, and the calendar store's `getEnabledGranularities()` all iterate or switch on the set of granularities. The type system would guide you through most of this, but the calendar components assume a month grid with weeks inside it — a quarter view would need new layout work.
- Changing the canonical key scheme. It's used in `byKey` indexing, `findAdjacent()` sorting, and `fileMapKey()` in the calendar. These are separate functions with the same contract; changing one without the other would silently break file lookups.

**Hard changes:**

- Supporting multiple files for the same date and granularity. The entire cache is built around a one-to-one mapping enforced by `set()`. The calendar's `FileMap` stores `TFile | null`, not an array. The `getPeriodicNote()` return type is `TFile | null`. This assumption is structural.
- Making the test suite import `cache.ts` or `template.ts` directly. These modules import from `obsidian` at the top level, and there is no mock for the Obsidian module in the test infrastructure. The test preload only provides `window.moment`. Issue #116 proposes extracting the pure logic from `template.ts` into a separate module; issue #93 tracks the drift risk of the current re-implementation approach. This is the system's most significant testing gap.
- Replacing Moment.js. The format string convention, the canonical key scheme, the `startOf()` normalization, the locale configuration, the template token parsing, the calendar date math — all depend on Moment's API surface. Moment is not wrapped in an abstraction layer; it's used directly everywhere including `window.moment` as a global.

## Uncertainties and tensions

**The `getPeriodicNotes()` method is puzzling.** It scans all `byPath` entries linearly to find notes within a time range, optionally including finer granularities. This is the only public cache method that doesn't use the O(1) index. The calendar's `computeFileMap()` doesn't call it — it calls `getPeriodicNote()` (singular) per cell. I cannot find a caller for `getPeriodicNotes()` in the current codebase. It may exist for external plugin consumers or may be dead code left from a previous design.

**The `fileMapKey()` and `canonicalKey()` functions compute the same concept differently.** `canonicalKey()` in cache.ts uses `date.startOf(granularity).toISOString()`. `fileMapKey()` in store.ts uses `date.format(DEFAULT_FORMAT[granularity])`. For day/month/year these produce equivalent uniqueness (different string representations of the same partition). For weeks, `toISOString()` after `startOf("week")` gives the locale-aware week start as an ISO timestamp, while `DEFAULT_FORMAT.week` is `"gggg-[W]ww"` which gives the locale-aware week number. These should be equivalent partitions — the same dates should collide — but they're not the same strings, and they're maintained independently. A locale where `startOf("week")` and `gggg-[W]ww` disagree would produce a subtle lookup miss between the cache and the calendar. I believe Moment keeps these consistent, but the system relies on an implicit invariant that is nowhere tested or documented.

**The `createPeriodicNote()` path in main.ts renders the template eagerly, but the cache's `resolve()` path also triggers template application for zero-byte files.** When `openPeriodicNote()` calls `createPeriodicNote()`, the template is read, rendered, and written as the file content. Then `vault.create()` fires a `create` event, the cache resolves the file, sees `file.stat.size === 0`, and tries to apply the template again. But the file is no longer zero bytes because `vault.create()` was called with the rendered content. So the cache's template trigger is a no-op for the normal creation path. It exists for the _other_ creation path: when a user creates a file manually (or another plugin creates it) with the right name in the right folder, empty. The cache picks it up, recognizes it as a periodic note, and fills in the template. This is clever but the two paths could diverge if someone changed only one.

**Settings loading is defensive but not migrating.** `loadSettings()` clones `DEFAULT_SETTINGS`, then merges any saved granularity configs on top with spread. This means new settings fields get defaults automatically, and removed fields are silently dropped. But it also means if a v1 user upgrades to v2, their settings shape (`saved?.granularities` check) likely fails and they get fresh defaults. The v2 design spec says this is intentional ("one user — no migration path needed"), but the code doesn't warn or log when this happens.

**The `isGranularityEnabled()` method in CalendarStore defaults to `true` for day granularity when settings are missing** (`?? granularity === "day"`, line 62). This fallback doesn't exist anywhere else in the codebase — in the cache, commands, and settings, a disabled granularity is simply skipped. The calendar apparently wants to always show day cells even if the day granularity is technically disabled in settings. This might be intentional (a calendar without day cells is useless) or might be a leftover guard from before settings loading was reliable.

**The v2 refactor was a single-author, big-bang rewrite.** The commit history from `a1e4a5a` through `0ba5d35` shows a carefully sequenced but monolithic refactor: types first, then constants, then format, then template, cache, commands, settings, main, calendar updates, old module deletion, test rewrites. The design spec was committed, reviewed, amended, then deleted after completion. The spec listed issues it would close (#85, #93, #100, #102, #110). Of those, #93 (test re-implementation drift) and #102 (Svelte store migration) remain open — the refactor addressed them partially but not fully. The remaining open issues (#115, #116, #117, #118) are all optimization and testing work that the refactor deferred. The codebase is in a stable post-refactor state, but the testing boundary problem — the inability to import `cache.ts` and `template.ts` in tests — is the most significant piece of unfinished structural work.
